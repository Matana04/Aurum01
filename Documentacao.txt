model Usuario {
  id        String    @id @default(uuid())
  nome      String
  email     String    @unique
  senha     String    // Lembre-se de usar bcrypt para gerar o hash da senha no backend
  
  // Relacionamentos
  despesas  Despesa[]
  metas     Meta[]

  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt

  @@map("usuarios") 
}

model Despesa {
  id        String   @id @default(uuid())
  titulo    String
  valor     Decimal  @db.Decimal(10, 2) // Perfeito para dinheiro: até 10 dígitos, 2 casas decimais
  data      DateTime 
  
  // Chave Estrangeira
  usuarioId String
  usuario   Usuario  @relation(fields: [usuarioId], references: [id], onDelete: Cascade)

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@map("despesas")
}

model Meta {
  id         String   @id @default(uuid())
  titulo     String
  valor      Decimal  @db.Decimal(10, 2)
  dataInicio DateTime
  dataFinal  DateTime
  
  // Chave Estrangeira
  usuarioId  String
  usuario    Usuario  @relation(fields: [usuarioId], references: [id], onDelete: Cascade)

  createdAt  DateTime @default(now())
  updatedAt  DateTime @updatedAt

  @@map("metas")
}








Serei bem direto: para um sistema financeiro e pensando no longo prazo que discutimos antes, **eu prefiro e recomendo fortemente o PostgreSQL.**

O MySQL é um banco de dados fantástico (e o mais usado no mundo), mas o PostgreSQL é amplamente considerado o "padrão ouro" da indústria quando o assunto é lidar com dinheiro, integridade rigorosa e consultas complexas.

Aqui estão os motivos técnicos pelos quais o Postgres leva vantagem no seu cenário:

### 1. Rigor e Integridade de Dados

Em finanças, você quer que o banco de dados seja o mais "chato" possível. O MySQL às vezes tenta ser "amigável" demais e pode, dependendo da configuração, truncar dados ou fazer conversões silenciosas se algo der errado. O PostgreSQL é estrito: se um dado não bate perfeitamente com a regra, ele trava a transação e devolve um erro claro. Para o seu dinheiro, isso é uma proteção vital.

### 2. Particionamento de Tabelas (O Segredo do Longo Prazo)

Lembra que conversamos sobre o alto volume de dados gerado por cada usuário ao longo dos anos? O PostgreSQL possui um sistema nativo de **Table Partitioning** muito mais maduro que o do MySQL. Quando sua tabela de `Despesas` chegar a 50 milhões de linhas, o Postgres permite dividi-la "por debaixo dos panos" (ex: por ano) de forma muito eficiente, mantendo as buscas do Prisma extremamente rápidas.

### 3. Concorrência (MVCC)

Sistemas financeiros precisam lidar muito bem com concorrência (ex: uma cobrança automática e uma compra no cartão acontecendo no exato mesmo milissegundo). O PostgreSQL usa um sistema chamado MVCC (*Multiversion Concurrency Control*) de forma brilhante, garantindo que as leituras e escritas simultâneas não bloqueiem umas às outras, evitando o famoso erro de "gasto duplo".

---

### Resumo do Comparativo

| Característica | PostgreSQL | MySQL |
| --- | --- | --- |
| **Foco Principal** | Integridade, conformidade e dados complexos | Velocidade de leitura, simplicidade e web |
| **JSON e Metadados** | Excelente (o tipo `JSONB` é imbatível) | Bom, mas menos otimizado |
| **Particionamento** | Excelente (Ideal para o seu alto volume) | Funcional, mas com mais limitações |
| **Hospedagem Moderna** | Excelente (Supabase, Vercel, Neon) | Excelente (PlanetScale, AWS, Aiven) |

### O veredito para você:

A boa notícia é que **o Prisma esconde 90% dessas diferenças de você**. O código que você vai escrever no seu backend em Node/TypeScript será literalmente idêntico para ambos.

* Se você **já tem muita experiência com MySQL** e se sente seguro para gerenciá-lo, pode manter. O motor InnoDB do MySQL é totalmente capaz de rodar um sistema financeiro.
* Se você **está livre para escolher**, vá de **PostgreSQL**. É a escolha padrão das *fintechs* modernas (como Nubank, Stripe, etc.) por um bom motivo.

**Qual dos dois faz mais sentido para o seu nível de conforto atual?** Se quiser mudar para o PostgreSQL, a única coisa que precisamos fazer no seu `schema.prisma` é trocar a linha `provider = "mysql"` para `provider = "postgresql"`. Posso fazer isso ou já podemos pular para a criação das rotas do CRUD!
